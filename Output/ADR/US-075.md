**ADR: Spatial Gradient Palette Module (`src/visual/spatialGradient.ts`)**

**Status:** Accepted

**Context:** US-075 introduces position-based vertex coloring for terrain. Rather than computing colors purely from uniforms in the shader (the current pattern), this feature pre-computes per-vertex RGB colors on the CPU based on world-space position and a seeded gradient palette. This enables multi-stop color gradients across geometry surfaces.

**Decision:** Create a new shared module `src/visual/spatialGradient.ts` that exposes:
- `SpatialGradientPalette` interface (3-5 color stops with positions)
- `createSpatialGradient()` — deterministic gradient generation from seed + palette params
- `sampleGradient()` — interpolation at a normalized parameter
- `computeVertexColors()` — bulk position-to-color mapping for Float32Array geometry data

The module is opt-in: systems attach an `aVertexColor` attribute and set `uHasVertexColor = 1.0`. Shaders use `mix()` to blend between the existing uniform-color path (when `uHasVertexColor = 0.0`) and the vertex-color path.

**Consequences:**
- New shared interface available for future systems (cube lattice, fractal growth) to adopt spatial gradients without shader changes.
- Pre-computed colors are static at init time and do not follow `evolution.ts` palette drift — this is an intentional artistic choice for gradient structure stability.
- Attribute name `aVertexColor` (vs existing `color` convention) chosen to distinguish spatial-gradient colors from per-point hue-offset colors used by other systems.
- The `uHasVertexColor` uniform guard follows the established `uHasSizeAttr` pattern for optional attribute fallback.
